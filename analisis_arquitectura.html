Arquitectura del Sistema: Trendicore

Documento Técnico de Diseño y Patrones Estructurales de Software

1. Resumen Ejecutivo

La aplicación Trendicore ha sido conceptualizada, diseñada y refactorizada bajo los principios de la Arquitectura Limpia (Clean Architecture) combinada con patrones de Arquitectura Hexagonal (Puertos y Adaptadores).

El objetivo arquitectónico primordial es garantizar un sistema altamente desacoplado. En este diseño, la lógica de negocio central o de dominio es completamente independiente de toda implementación periférica técnica, tal como las interfaces de usuario (Next.js/React), las herramientas de persistencia y bases de datos (Supabase) y dependencias de terceros.

El código establece barreras claras mediante las siguientes capas principales:

  • Dominio (Entities): Contiene las reglas y abstracciones de negocio puras (e.g., Product, User, Permissions).
  • Aplicación (Use Cases): Aloja la orquestación de operaciones de la aplicación (e.g., CreateProductUseCase, CheckPermissionUseCase).
  • Infraestructura (Adapters): Aloja implementaciones que concretan puertos o interfaces definidos, interactuando con agentes externos (e.g., SupabaseProductRepository).
  • Presentación (UI): Comprende Componentes React interactivos y APIs manejadas puramente por rutado (App Router).

3. Justificación de Refactorización

Fase I (Legacy/AI-Generated): La aplicación prototipada inicialmente mediante IA generativa funcionaba de forma monolítica. Es decir: la obtención de datos, el enrutado, la pintura de UI y las restricciones de base de datos residían conjuntamente dentro de las páginas web (Server Components o API Routes sobrecargadas). Esto redujo la calidad arquitectónica asumiendo gran acumulación de deuda técnica y bloqueando una cobertura seria en validaciones de tests unitarios (Test-Driven Development).

Ventajas Adquiridas con TDD y Arquitectura Limpia:

  • Incremento de Testabilidad (Intercambiabilidad de Componentes): Al purificar las reglas de negocio en la capa /core e introducir puertos con interfaces para los repositorios, resulta trivial pasar "Mocks Repository" durante test. Los métodos de LoginUseCase pueden comprobarse al 100% aislando totalmente el servidor físico de Supabase o las caídas de red.
  • Inversión de Control y Desacoplamiento (Principios SOLID): Componentes visuales robustos se dedican puramente a UI sin contaminarse ejecutando lógica de SQL o lógicas complejas estáticas de si un producto es publicable o no.
  • Escalabilidad Superior y Extensibilidad: Alterar un servicio por otro (ejemplo, rotar de Supabase a un backend custom Node/Postgres u otro BaaS) se localiza creando exclusivamente un nuevo adapter en /infrastructure sin alterar los scripts principales de los UseCases o la Entidad abstracta.

4. Detalle de Componentes y Responsabilidades de Módulos

El código central del sistema se reparte geográficamente según los siguientes focos de responsabilidad.

Módulos de Front-End (UI / Next.js)

  • Capa /app (Enrutador Web): Gobierna estrictamente las divisiones frontales para el Server Side Rendering y Client Routing de Next.js ((public) y (admin)). Administra layouts globales de forma limpia.
  • Capa /components (Presentación Visual): Estanca los assets y utilidades del modelo abstracto de UI React. Los componentes como ProductCard y ProductForm son "Dumb y Presentacionales". Operan a partir de inyección de properties sin lógicas de infra.
  • Capa /hooks & /providers: Proporcionan conectores limpios de contexto React hacia los features visuales o de optimización UX (i18n, Wishlist, Autenticación Client-Side).

Módulos Back-End (Core & Domain)

  • Entities (/domain): El santuario puro de la aplicación. Modelos vitales auto-contenidos, aislados estructuralmente como Product.js y User.js, garantizando verificaciones de sintaxis y estado antes de accionar operaciones.
  • Use Cases (/application): Modos directos de interacciones (Agentes Activos). Representan acciones transaccionales del ciclo de software: UpdateProductUseCase.js o LogoutUseCase.js orquestando entidades sobre los repositorios.
  • Adapters (/infrastructure): Clases pragmáticas concretadas (SupabaseProductRepository). Usan el objeto cliente físico supabaseClient e interactúan en el protocolo bruto (SQL, Rest).
  • Controladores de Red (/app/api): Funciones exponiendo endpoints REST aislados para conectar web remota hacia el Core App y devolver JSON estándar con estado nativo HTTP.
