Arquitectura del Sistema: Trendicore

Documento Técnico de Diseño y Patrones Estructurales de Software

1. Resumen Ejecutivo

La aplicación Trendicore ha sido conceptualizada, diseñada y refactorizada bajo las filosofías puras de Clean Architecture (Arquitectura Limpia) y Arquitectura Hexagonal (Puertos y Adaptadores).

El objetivo arquitectónico primordial es garantizar un sistema blindado y altamente tolerante al cambio estructural. En este diseño, la lógica de negocio central (El Dominio) es completamente agnóstica de toda implementación periférica tecnológica, tal como el renderizado gráfico (Next.js/React), las herramientas de persistencia y bases de datos (Supabase) y dependencias de red de terceros.

El código establece fronteras infranqueables y estrictas mediante las siguientes capas principales:

  • Dominio (Entities y Errors): Contiene las entidades puras (e.g., Product) y las abstracciones de negocio mediante el novedoso sistema semántico de DomainError y sus hijos directos (DuplicateProductError, ProductNotFoundError).
  • Aplicación (Use Cases): Aloja los controladores directos o flujos de usuario (e.g., CreateProductUseCase). Son el pegamento que utiliza puertos para comunicarse con persistencias sin importarle cómo funcionan por debajo.
  • Infraestructura (Adapters): Aloja implementaciones que concretan puertos definidos mediante JSDoc estricto (IProductRepository), interactuando con agentes externos (como es el caso específico de SupabaseProductRepository).
  • Presentación (UI): Comprende componentes funcionales y de interfaz gráfica interactivos junto al motor de SSR de App Router (Next.js), enteramente divorciados de la Base de Datos.

3. Justificación de Refactorización Integral

Fase I (Prototipo Legacy): La aplicación inicial se estructuró a partir del código sugerido automáticamente por modelos LLM genéricos de IA, priorizando velocidad de lanzamiento. Esto desembocó en acoplamientos peligrosos "Espagueti" donde funciones de Bases de Datos compartían un mismo bloque de código JSX con Botones en pantalla, obstaculizando escalar al estar atrapado todo en Server Components sobrecargados. Al mismo tiempo no existía un sistema unificado de excepciones si el backend devolvía código PGRST116 (desconocido e intimidador).

Ventajas de la Fase II (Arquitectura de Élite y TDD):

  • Incremento Extremo en Testabilidad: Al aislar todo rastro de software foráneo detrás de IProductRepository y IPermissionRepository (documentados de facto vía Intellisense y JSDoc paramétrico), los Unit Tests como el del PermissionManager corren en memoria a velocidad luz sin tener que instanciar mock-ups pesados o simuladores de red para emular la Base de datos de Supabase.
  • Gestión de Errores Seguros: El frontend ya no tiene que parsear errores técnicos de PostgreSQL (ej. 'Violación 23505'). El componente SupabaseProductRepository atrapa estos fallos y escupe limpiamente excepciones semánticas como DuplicateProductError heradadas de DomainError hacia arriba, habilitando pop-ups amigables con el usuario instantáneamente.
  • Agilidad de Reemplazo Institucional: Al regirse estrictamente por interfaces abstractas (Contratos), tu equipo podría desarrollar el día de mañana un MongoDbProductRepository o un archivo MemoryMockRepository y lo único que deben hacer en CreateProductUseCase es pasarlo como objeto en su constructor. La lógica jamás notará la diferencia.

4. Detalle de Componentes Modulares por Capas

El territorio estructural de Trendicore se mapea siguiendo responsabilidades de alcance geográfico exactas. Nada se comunica o mezcla en horizontal de forma aleatoria.

Módulos Físicos Front-End (Next.js)

  • Capa /app (Enrutamiento Híbrido): Gobierna el file-system routing dividiéndose magistralmente en secciones seguras mediante Grupos de Lógica entre (public), (admin) y sus respectivos layout.js.
  • Capa /components (Lógica Visual Estúpida "Dumb"): Alberga componentes atómicos desacoplados. Componentes como Navbar, botones o campos de texto son genéricos a todo contexto. Son orquestados a nivel vista sin llevar pesos pesados de estado o llamadas DB.
  • Capa /hooks & /providers: Proporcionan conectores limpios de contexto React que nutren parámetros y memorias transversales al usuario (e.g. LocaleProvider, Carrito Global y useProducts.js como fetcher cliente).

Módulos Físicos Back-End (Hexagonal Core)

  • Dominio (/core/shared/domain y derivados): Son el "Santo Grial". Archivos absolutos como Product.js, DomainError.js o ProductErrors.js. Modelos vitales auto-contenidos e impenetrables.
  • Aplicación (/core/product/application): Operadores del negocio (Ej. TrackClickUseCase o DeleteProductUseCase). Encapsulan el comportamiento real del ciclo lógico controlando a los Repositorios y produciendo respuestas estructuradas.
  • Adaptadores Infra (/core/product/infrastructure/adapters): Componentes pragmáticos invasivos como el SupabaseProductRepository.js conectados al "Mundo Exterior". Transforman la jerga extraña del cliente físico en DTOs o instancias perfectas de tu Product de Dominio.
  • Puertos (Controllers/APIs): Rutas como /api/admin/products actúan simplemente como un peaje traduciendo JSON del frontend a parámetros planos para los Casos de Uso del core y devolviendo Códigos HTTP nativos estándar.
